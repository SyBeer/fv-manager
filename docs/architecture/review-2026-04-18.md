# Analiza architektoniczna — FV Manager

## Scenariusze biznesowe

### Dla kogo?
System jest narzędziem **prosumenta indywidualnego** — właściciela instalacji fotowoltaicznej, który:
- posiada inwerter SolarEdge (integracja potwierdzona w `project.json`)
- rozlicza się z PGE w modelu net-meteringu (współczynnik 0.8 zakodowany w `calculations.py`)
- eksploatuje pojazd elektryczny (moduł EV z kalkulatorem oszczędności vs spalinowe)
- korzysta z Home Assistant jako platformy domowej automatyki

### Jaka wartość biznesowa?
| Wartość | Realizacja w systemie |
|---|---|
| Monitoring finansowy inwestycji | ROI tracking + projekcja break-even |
| Optymalizacja kosztów energii | Kalkulacja autokonsumpcji vs oddanie do sieci |
| Analiza porównawcza EV vs ICE | `calc_ev_savings()` z cenami paliw |
| Cyfryzacja danych z faktur | Import z Excela SolarEdge/PGE |

### Cele strategiczne (perspektywa prosumenta)
1. **Weryfikacja opłacalności** — czy inwestycja w PV się zwraca i kiedy
2. **Optymalizacja autokonsumpcji** — minimalizacja energii oddawanej do sieci
3. **Całościowy bilans energetyczny** — PV + EV + sieć w jednym miejscu

## Ocena strategiczna

### Alignment architektoniczny (TOGAF — Architecture Vision)

**Warstwa biznesowa (Business Architecture)**

System realizuje jeden, dobrze zdefiniowany bounded context — *zarządzanie efektywnością instalacji PV*. Granica domeny jest czytelna. Problem pojawia się przy **rozszerzeniu na EV** — to odrębna subdomena (transport) wciśnięta do tej samej aplikacji bez separacji:

```
src/services/calculations.py
└── calc_monthly()       ← domena PV ✅
└── calc_roi()           ← domena PV ✅
└── calc_ev_savings()    ← domena Transport ⚠️ — naruszenie SRP na poziomie modułu
```

**Warstwa danych (Data Architecture)**

Pojedynczy plik `fv.db` (SQLite) z pięcioma tabelami. Dla obecnej skali — właściwy wybór. Krytyczny problem: **brak migracji schematu**. `init_db()` używa `CREATE TABLE IF NOT EXISTS`, co oznacza, że zmiany schematu (np. nowe kolumny) nie są automatycznie aplikowane w istniejących instalacjach.

```python
# db.py — brak wersjonowania schematu
await db.executescript("""
    CREATE TABLE IF NOT EXISTS readings (...)  # ← co jeśli schemat się zmieni?
""")
```

**Warstwa aplikacyjna (Application Architecture)**

```
FastAPI (main.py)
├── Routing — ✅ zrozumiałe endpointy REST/HTML
├── Business Logic — ⚠️ częściowo w main.py (logika wzbogacania odczytów)
├── Data Access — ⚠️ bezpośrednie zapytania SQL w handlerach
└── Templates — ✅ poprawna separacja Jinja2
```

Handlery w `main.py` zawierają logikę wzbogacania danych (pętla `enrich`), która powinna znajdować się w warstwie serwisów:

```python
# main.py — logika biznesowa w handlerze (antywzorzec)
for r in readings:
    c = calc_monthly(r["production_kwh"], ...)
    enriched.append({**r, **c})
```

**Warstwa techniczna (Technology Architecture)**

| Komponent | Wybór | Ocena |
|---|---|---|
| Framework | FastAPI | ✅ właściwy dla async I/O |
| Baza danych | SQLite + aiosqlite | ✅ odpowiedni dla skali |
| Templating | Jinja2 | ✅ wystarczający |
| CSS | Tailwind CDN | ⚠️ CDN = zależność sieciowa, brak offline |
| Wykresy | Chart.js CDN | ⚠️ j.w. |
| Testy | pytest + httpx | ⚠️ tylko unit testy kalkulacji, brak integracyjnych |

### Ryzyka strategiczne

**RYZYKO 1 — Hardkodowany model regulacyjny (WYSOKIE)**
```python
# calculations.py linia ~17
net_metering_pool = sent_to_grid * 0.8  # ← współczynnik net-meteringu zakodowany
```
Polska przeszła z net-meteringu na net-billing w 2022 roku. Nowe instalacje działają w innym modelu rozliczeń. System obsługuje wyłącznie stary model, co ogranicza jego użyteczność i może generować błędne kalkulacje dla instalacji po 2022 roku. **To największe ryzyko merytoryczne.**

**RYZYKO 2 — Brak uwierzytelnienia (WYSOKIE dla deploymentu)**
Aplikacja nie posiada żadnego mechanizmu autoryzacji. `project.json` wskazuje na integrację z Home Assistant — jeśli HA jest wystawiony na internet, FV Manager dostępny w sieci lokalnej bez auth to wektor ataku.

**RYZYKO 3 — Import z Excela jako jedyne źródło danych (ŚREDNIE)**
```python
# importer.py
ws = wb["Moja instalacja"]  # ← zakodowana nazwa arkusza
for row in ws.iter_rows(min_row=5, max_row=200, ...):
    production = row[3]  # ← zakodowane indeksy kolumn
```
Zmiana formatu eksportu przez SolarEdge lub PGE złamie import. Brak walidacji struktury pliku.

**RYZYKO 4 — Brak audytu i historii zmian (NISKIE-ŚREDNIE)**
Operacja usunięcia odczytu (`/odczyty/{id}/usun`) jest nieodwracalna. Brak soft-delete, brak logu zmian.

## Luki i rekomendacje EA

### Luka 1 — Brak obsługi net-billingu (GAP KRYTYCZNY)

**Rekomendacja:** Wprowadzić abstrakcję modelu rozliczeń jako strategię (wzorzec Strategy):

```python
# Proponowana architektura
class SettlementModel(Protocol):
    def calc_savings(self, production, sent, taken, price) -> dict: ...

class NetMeteringModel:      # stare instalacje (przed 2022)
    COEFFICIENT = 0.8
    ...

class NetBillingModel:       # nowe instalacje (po 2022)
    def calc_savings(self, ...):  # cena dynamiczna za oddaną energię
        ...
```

W bazie danych dodać pole `settlement_model` w tabeli `investments`, powiązać z kalkulatorem.

### Luka 2 — Brak Repository Pattern (GAP ARCHITEKTONICZNY)

**Rekomendacja:** Wydzielić warstwę dostępu do danych:

```
src/
├── repositories/
│   ├── readings_repo.py    # get_all(), get_by_period(), create(), delete()
│   ├── investments_repo.py
│   └── ev_repo.py
├── services/
│   ├── calculations.py     # czysta logika biznesowa (bez DB)
│   └── reporting.py        # agregacje i raporty
└── main.py                 # tylko routing + DI
```

To odblokuje testowanie integracyjne bez bazy danych i uprości `main.py`.

### Luka 3 — Brak migracji schematu (GAP OPERACYJNY)

**Rekomendacja:** Wprowadzić Alembic lub minimalny własny mechanizm wersjonowania:

```python
# Minimalny mechanizm bez Alembic
SCHEMA_VERSION = 3

async def migrate_db(db):
    version = await get_schema_version(db)
    if version < 2:
        await db.execute("ALTER TABLE readings ADD COLUMN ev_kwh REAL")
    if version < 3:
        await db.execute("ALTER TABLE investments ADD COLUMN settlement_model TEXT DEFAULT 'net_metering'")
    await set_schema_version(db, SCHEMA_VERSION)
```

### Luka 4 — Brak integracji API (GAP CAPABILITY)

System deklaruje integracje z SolarEdge i PGE, ale realizuje je wyłącznie przez import pliku Excel — co jest procesem manualnym. 

**Rekomendacja (roadmapa):**
```
Faza 1 (obecna):  Excel Import          ← zrealizowane
Faza 2:           SolarEdge API polling ← SolarEdge ma publiczne API REST
Faza 3:           Home Assistant sensor ← dane przez MQTT lub HA REST API
Faza 4:           PGE e-faktura parser  ← automatyczny import faktur PDF/XML
```

SolarEdge udostępnia endpoint `/site/{siteId}/energy` — integracja wyeliminowałaby główną przyczynę błędów (ręczny eksport/import).

### Luka 5 — Brak zewnętrznych cen energii (GAP DANYCH)

Cena `DEFAULT_PRICE_KWH=0.75 zł` w `.env` jest statyczna. Ceny energii dla prosumentów zmieniają się kwartalnie.

**Rekomendacja:** Podłączyć API URE (Urząd Regulacji Energetyki) lub agregator cen energii jako źródło referencyjne. Alternatywnie — mechanizm alertu gdy `price_per_kwh` w odczytach różni się >20% od domyślnej.

### Luka 6 — Brak strategii offline/resilience (GAP INFRASTRUKTURALNY)

Tailwind i Chart.js ładowane z CDN — przy braku internetu (np. awaria routera w inteligentnym domu) interfejs się rozjeżdża lub nie ładuje wykresów.

**Rekomendacja:** Vendoryzacja zależności frontendowych do katalogu `static/`. Priorytet niski dla użytku domowego, wysoki jeśli planowany jest deployment produkcyjny.

## 2. Zidentyfikowane Problemy Architektoniczne

### P1 — Brak separacji warstw (KRYTYCZNY)
```python
# main.py — jeden handler robi WSZYSTKO:
async def dashboard(request: Request):
    db = await get_db()           # ← infrastruktura
    readings = await _get_readings(db)  # ← dostęp do danych
    roi = calc_roi(readings, ...)  # ← logika biznesowa
    return templates.TemplateResponse(...)  # ← prezentacja
```

### P2 — Zarządzanie połączeniem DB (POWAŻNY)
```python
# Wzorzec try/finally w każdym handlerze — brak dependency injection
db = await get_db()
try:
    ...
finally:
    await db.close()
```
Brak `Depends()` z FastAPI powoduje ryzyko niezamkniętych połączeń przy wyjątkach przed blokiem `try`.

### P3 — Brak walidacji danych wejściowych
Formularz `reading_form.html` używa `pattern="\d{4}\.\d{2}"` tylko po stronie HTML — brak walidacji Pydantic po stronie serwera.

### P4 — Hardkodowane stałe biznesowe
```python
net_metering_pool = sent_to_grid * 0.8  # wskaźnik net-meteringu zahardkodowany
```

### P5 — Brak obsługi błędów HTTP
Żaden handler nie zwraca poprawnych kodów HTTP przy błędach (404, 422, 500).

## 4. ADR — Architecture Decision Records

---

**ADR-001: Wzorzec Repository dla dostępu do danych**
- **Kontekst:** Obecnie zapytania SQL są rozrzucone po `main.py` w helperach `_get_readings()`, `_get_investments()`. Utrudnia to testowanie i powoduje duplikację logiki zapytań.
- **Decyzja:** Wprowadzić wzorzec Repository — klasy `ReadingRepository` i `InvestmentRepository` z metodami `get_all()`, `get_by_id()`, `create()`, `update()`, `delete()`. Repozytoria przyjmują `aiosqlite.Connection` przez konstruktor.
- **Konsekwencje:**
  - ✅ Możliwość mockowania w testach bez trafienia w bazę
  - ✅ Jeden punkt zmiany przy migracji z SQLite na PostgreSQL
  - ✅ Eliminacja duplikacji SQL w handlerach
  - ❌ Dodatkowe ~4 pliki i warstwa abstrakcji dla małego projektu
  - ❌ Konieczność refactoringu istniejących handlerów

---

**ADR-002: FastAPI Depends() dla zarządzania połączeniem DB**
- **Kontekst:** Obecny wzorzec `db = await get_db() / try / finally` jest powtarzany w każdym handlerze (>10 razy). Ryzyko wycieku połączenia przy wyjątku przed blokiem `try`.
- **Decyzja:** Przepisać `get_db()` jako async generator używany przez `Depends()`:
  ```python
  async def get_db():
      db = await aiosqlite.connect(DB_PATH)
      db.row_factory = aiosqlite.Row
      try:
          yield db
      finally:
          await db.close()
  
  # W handlerze:
  async def dashboard(request: Request, db=Depends(get_db)):
      ...
  ```
- **Konsekwencje:**
  - ✅ Gwarantowane zamknięcie połączenia przez FastAPI lifecycle
  - ✅ Eliminacja ~30 linii boilerplate w main.py
  - ✅ Łatwy override w testach (`app.dependency_overrides`)
  - ❌ Wymaga znajomości mechanizmu Depends() przez zespół

---

**ADR-003: Pydantic schemas dla walidacji danych wejściowych**
- **Kontekst:** Formularze HTML walidują dane tylko po stronie klienta (atrybut `pattern`). Brak walidacji serwerowej przy POST `/odczyty/nowy` i `/inwestycje/nowa` — możliwe zapisy nieprawidłowych danych.
- **Decyzja:** Dodać schematy Pydantic (`ReadingCreate`, `InvestmentCreate`) i walidować dane z formularzy przed zapisem do DB. Przy błędzie walidacji zwracać HTTP 422 z komunikatem.
- **Konsekwencje:**
  - ✅ Ochrona integralności danych w bazie
  - ✅ Czytelna dokumentacja kontraktu danych (auto-OpenAPI)
  - ✅ Reużywalność schematów w potencjalnym REST API
  - ❌ Formularze HTML wymagają dodatkowej obsługi błędów w szablonach

---

**ADR-004: Ekstrakcja stałych biznesowych do konfiguracji**
- **Kontekst:** Współczynnik net-meteringu `0.8` jest zahardkodowany w `calculations.py`. Zmiana regulacji (np. przejście na net-billing) wymaga edycji kodu. Plik `.env.example` zawiera już `DEFAULT_PRICE_KWH` — precedens istnieje.
- **Decyzja:** Przenieść stałe biznesowe do `.env` / `Settings` (Pydantic BaseSettings):
  ```
  NET_METERING_RATIO=0.80
  DEFAULT_PRICE_KWH=0.75
  ```
  `calc_monthly()` przyjmuje `net_metering_ratio` jako parametr z domyślną wartością z Settings.
- **Konsekwencje:**
  - ✅ Zmiana reguł rozliczania bez modyfikacji kodu
  - ✅ Różne konfiguracje dla różnych taryf
  - ❌ Jeden dodatkowy parametr w sygnaturze funkcji (łamie obecne testy)

---

**ADR-005: Modularyzacja routerów FastAPI**
- **Kontekst:** `main.py` zawiera ~300+ linii z wszystkimi endpointami (dashboard, odczyty, ROI, EV, inwestycje, import). Plik rośnie z każdą funkcją.
- **Decyzja:** Podzielić na osobne `APIRouter` w `src/api/routers/`. Każdy moduł odpowiada jednej domenie. `main.py` tylko rejestruje routery przez `app.include_router()`.
- **Konsekwencje:**
  - ✅ Każdy router testowalny niezależnie
  - ✅ Łatwe dodawanie nowych modułów (np. `alerts.py`)
  - ✅ Możliwość przypisania prefixów i tagów OpenAPI per-router
  - ❌ Refactoring istniejącego kodu — ryzyko regresji bez testów integracyjnych

## 5. NFR — Non-Functional Requirements

| **Wydajność** | Import pliku Excel | < 5s dla pliku z 200 wierszami | Brak pomiaru |
| **Wydajność** | Czas odpowiedzi strony ROI | p95 < 300ms (obliczenia skumulowane) | Brak pomiaru |

## Kluczowe rekomendacje bezpieczeństwa

### 🔴 Krytyczne (natychmiastowe działanie)

1. **[AUTENTYKACJA]** Wdrożyć uwierzytelnianie HTTP Basic Auth jako minimum — jedna linijka middleware w FastAPI zatrzymuje nieautoryzowany dostęp. Dane osobowe (faktury, tokeny) są teraz publicznie dostępne dla każdego w sieci.

```python
# Minimum viable auth — FastAPI middleware
from fastapi import Depends, HTTPException, status
from fastapi.security import HTTPBasic, HTTPBasicCredentials
import secrets, os

security = HTTPBasic()
def require_auth(credentials: HTTPBasicCredentials = Depends(security)):
    ok = secrets.compare_digest(credentials.username, os.environ["APP_USER"]) and \
         secrets.compare_digest(credentials.password, os.environ["APP_PASS"])
    if not ok:
        raise HTTPException(status_code=status.HTTP_401_UNAUTHORIZED,
                          headers={"WWW-Authenticate": "Basic"})
```

2. **[TOKEN HOME ASSISTANT]** Token HA przechowywany plaintext w SQLite to krytyczny błąd — kompromitacja pliku `fv.db` (backup, błędne uprawnienia) daje pełny dostęp do infrastruktury smart home. Zaszyfrować natychmiast lub przenieść do `.env`.

3. **[WALIDACJA URL HA]** Pole `ha_url` z bazy danych trafia bezpośrednio do klienta HTTP — możliwy SSRF (Server-Side Request Forgery) umożliwiający skanowanie sieci wewnętrznej.

# W endpoincie /import
MAX_SIZE = 10 * 1024 * 1024  # 10 MB

with tempfile.NamedTemporaryFile(suffix=".xlsx", delete=False) as tmp:

- **Kierunek:** Pull — dane o rachunkach (nr faktury `row[16]`, kwota `row[26]`, cena `row[21]`) są częścią arkusza `Moja instalacja`

- **Obsługa błędów:** Żadna dla danych finansowych — `price_per_kwh = row[21]` przyjmuje `None` bez ostrzeżenia; `invoice_gross = row[26]` nie jest walidowany
- **Ryzyko:**
  - Dane PGE (ceny, faktury) są zmieszane z danymi produkcyjnymi SolarEdge w jednym pliku — **brak separacji odpowiedzialności**
  - `price_per_kwh = None` propaguje się cicho przez cały system (kalkulacje ROI używają `default_price = 0.75`)
  - Logika `net_metering` zakodowana na stałe (`* 0.8`) — zmiana taryfy wymaga zmiany kodu
- **Rekomendacja:**
  - Wydzielić logikę parsowania danych finansowych PGE do osobnej funkcji
  - Dodać ostrzeżenie/flagę gdy `price_per_kwh` jest `None` dla historycznych rekordów
  - Parametryzować współczynnik net-meteringu (obecna wartość `0.8` jako stała w `ev_settings` lub `.env`)

---

### Home Assistant

- **Wzorzec:** REST API (potencjalnie)
- **Kierunek:** Pull — z kodu w DB wynika: `ha_url`, `ha_token`, `ha_entity` przechowywane w tabeli `ev_settings`
- **Timeout/SLA:** **Nieznany** — brak jakiejkolwiek implementacji integracji w dostarczonym kodzie
- **Obsługa błędów:** **Nie istnieje** — pola są w bazie danych, ale brak endpointów wywołujących HA API
- **Ryzyko:**
  - Integracja zadeklarowana w `project.json` i bazie danych, ale **niezaimplementowana** — ryzyko false security (użytkownik może sądzić, że dane EV są synchronizowane)
  - Token HA (`ha_token`) przechowywany w SQLite bez szyfrowania — **ryzyko bezpieczeństwa**
  - Brak walidacji URL (`ha_url`) — możliwe SSRF jeśli endpoint zostanie zaimplementowany bez sanityzacji
- **Rekomendacja:**
  - Zaimplementować lub jawnie oznaczyć jako `TODO` w UI
  - Token przenieść do `.env` zamiast SQLite, lub przynajmniej zaszyfrować
  - Przy implementacji: dodać timeout (≤5s), circuit breaker, fallback do wartości ręcznych
  - Dodać endpoint testowy (`GET /ha/test`) weryfikujący połączenie przed zapisem konfiguracji

# Analiza Data Architecture — FV Manager

## Model danych

### Encje i relacje

| Encja | Klucz główny | Kluczowe atrybuty | Uwagi |
|---|---|---|---|
| `readings` | `id` INTEGER | `period` UNIQUE, `year`, `month`, `production_kwh`, `sent_to_grid_kwh`, `taken_from_grid_kwh`, `ev_kwh`, `price_per_kwh`, `invoice_number`, `invoice_gross` | Centralna encja systemu |
| `investments` | `id` INTEGER | `date`, `description`, `cost_pln`, `power_kwp`, `notes` | Nakłady inwestycyjne na instalację |
| `ev_settings` | `id=1` (singleton) | `efficiency_kwh_per_100km`, `fuel_consumption_l_per_100km`, `annual_km`, `ha_url`, `ha_token`, `ha_entity` | Pojedynczy wiersz konfiguracyjny |
| `fuel_prices` | `id` INTEGER | `date`, `price_per_liter`, `fuel_type`, `source` | Historia cen paliw |

### Relacje (kardynalność)

```
investments  ──── brak FK ────  readings
                                    │
                            (logiczne powiązanie
                            przez year/month/period)

ev_settings (1:1 singleton) ←→ readings.ev_kwh
fuel_prices  ──── brak FK ────  ev_settings.fuel_type
```

> ⚠️ **Brak formalnych kluczy obcych** — relacje są wyłącznie logiczne/aplikacyjne. SQLite obsługuje FK, ale nie są zdefiniowane.

### Pola pochodne (nigdy nie są persystowane)

Pola obliczane w locie przez `calc_monthly()`:
- `auto_consumption` = `production_kwh − sent_to_grid_kwh`
- `total_consumed` = `auto_consumption + taken_from_grid_kwh`
- `net_metering_pool` = `sent_to_grid_kwh × 0.8`
- `savings_kwh`, `savings_pln`

## Przepływ danych

### Strumień 1 — Import z Excela (PGE/SolarEdge)

```
Excel (.xlsx)
  └─ arkusz "Moja instalacja" (wiersze 5–200)
       │
       ▼ importer.py
  Parsowanie kolumn (stałe indeksy: row[3], row[4]...)
  Filtrowanie: production IS NOT NULL
             + sent_meter IS NOT NULL (tylko real data)
  Transformacja: period → year, month (split ".")
               : round(float, 3)
               : sent_calc (delta, nie narastająco)
       │
       ▼ POST /import → main.py
  INSERT OR IGNORE INTO readings (dedupl. po UNIQUE period)
       │
       ▼ SQLite fv.db
```

> ⚠️ **Ryzyko kruche parsowanie** — indeksy kolumn hardkodowane (`row[3]`, `row[21]`, `row[26]`). Zmiana formatu pliku Excel spowoduje ciche błędy lub wyjątki.

### Strumień 2 — Ręczne wprowadzanie odczytów

```
Użytkownik → formularz /odczyty/nowy
  │
  ▼ POST /odczyty/nowy → main.py
  Walidacja: period pattern \d{4}\.\d{2}
  INSERT INTO readings
  └─ SQLite fv.db
```

### Strumień 3 — Kalkulacje energetyczne (read-only)

```
SQLite fv.db → _get_readings()
  │
  ▼ calc_monthly() [per wiersz, w pamięci]
  auto_consumption, savings_kwh, savings_pln...
  │
  ▼ calc_roi() [agregacja wszystkich odczytów]
  total_savings_pln, remaining_to_roi, months_to_roi
  │
  ▼ Jinja2 templates → HTML response
```

> ⚠️ **Brak cache** — przy każdym żądaniu przeliczane są wszystkie rekordy od początku. Przy ~100 odczytach to pomijalny koszt, ale architektonicznie warto odnotować.

### Strumień 4 — EV savings

```
readings.ev_kwh + ev_settings + fuel_prices (latest)
  │
  ▼ calc_ev_savings() [per miesiąc]
  km_driven, fuel_cost_equivalent, electricity_cost, net_savings
  │
  ▼ template ev.html
```

> ⚠️ **Sprzężenie danych** — `ev_kwh` jest przechowywane w encji `readings`, co powoduje mieszanie dwóch domenowo odrębnych danych (fotowoltaika vs. transport).

### Strumień 5 — Integracja Home Assistant (planowana/częściowa)

```
HA API → ha_url + ha_token (ev_settings)
  │
  ▼ [brak implementacji w main.py — konfiguracja istnieje w DB]
  ha_entity → odczyt stanu licznika EV
```

## Rekomendacje Data Architecture

### 🔴 Priorytet KRYTYCZNY

**1. Szyfrowanie/separacja credentiali HA**
```python
# Obecnie: ha_token TEXT w ev_settings (SQLite plaintext)
# Rekomendacja: przenieść do .env lub zaszyfrować (Fernet/keyring)
HA_TOKEN = os.getenv("HA_TOKEN")  # nie w DB
```
Token Home Assistant daje pełny dostęp do inteligentnego domu — nie powinien być w pliku `.db`.

**2. Ochrona pliku `fv.db`**

Plik `data/fv.db` powinien być w `.gitignore` (zawiera dane finansowe i token HA). Brak weryfikacji czy tak jest.

### 🟠 Priorytet WYSOKI

**3. Eliminacja kruchych indeksów kolumn w importerze**
```python
# Obecne (kruche):
production = row[3]
price_per_kwh = row[21]

# Rekomendacja: mapowanie przez nagłówki kolumn
COLUMN_MAP = {"Produkcja [kWh]": "production_kwh", ...}
headers = [cell.value for cell in ws[4]]
```

**4. Separacja `ev_kwh` z encji `readings`**

```sql
-- Nowa tabela:
CREATE TABLE ev_readings (
    id INTEGER PRIMARY KEY,
    period TEXT NOT NULL REFERENCES readings(period),
    ev_kwh REAL NOT NULL,
    source TEXT  -- 'manual' | 'home_assistant'
);
```
Uzasadnienie: dane fotowoltaiczne i transportowe to odrębne domeny; obecne podejście utrudni np. obsługę wielu pojazdów.

**5. Dodanie kluczy obcych i indeksów**
```sql
PRAGMA foreign_keys = ON;

CREATE INDEX idx_readings_year_month ON readings(year, month);
CREATE INDEX idx_fuel_prices_date ON fuel_prices(date DESC);
```

### 🟡 Priorytet ŚREDNI

**6. Persystencja pól obliczanych (opcjonalna)**

Dla audytu historycznego warto rozważyć persystowanie `savings_pln` w momencie zapisu — zmiana `DEFAULT_PRICE_KWH` retroaktywnie zmienia historyczne "oszczędności", co może być mylące przy analizie ROI.

```sql
ALTER TABLE readings ADD COLUMN savings_pln_snapshot REAL;
```

**7. Wersjonowanie schematu bazy (migrations)**

Brak mechanizmu migracji — `CREATE TABLE IF NOT EXISTS` w `init_db()` nie obsługuje zmian schematu. Rekomendacja: `alembic` lub prosta tabela `schema_version`.

**8. Walidacja danych przy imporcie**

```python
# Brak walidacji wartości ujemnych, anomalii:
assert production >= 0, f"Ujemna produkcja w {period}"
assert sent_to_grid <= production, f"Oddano więcej niż wyprodukowano"
```

---

### 🟢 Priorytet NISKI

**9. Strategia retencji i backup**

Jedyny plik danych to `data/fv.db` — brak mechanizmu backup. Rekomendacja: eksport CSV/JSON przez endpoint `/export` + dokumentacja ręcznego backup.

**10. Typ `REAL` dla wartości finansowych**

SQLite `REAL` = IEEE 754 double → błędy zaokrągleń przy sumowaniu. Dla danych finansowych rozważyć przechowywanie jako `INTEGER` (grosze × 100) lub użyć `Decimal` w warstwie aplikacji (już częściowo robione przez `round(..., 2)`).

## Kluczowe problemy UX

### 🔴 Krytyczne

**1. Brak obsługi błędów formularzy**
W `reading_form.html` — po błędzie walidacji server-side (np. duplikat okresu) użytkownik dostaje przekierowanie bez kontekstu. Formularz z `period` jako text input (`pattern="\d{4}\.\d{2}"`) jest podatny na literówki, a błąd jest cichy.

**2. Tabele bez responsywności mobilnej**
`readings.html` ma 9 kolumn, `ev.html` — 7 kolumn. Na telefonie lub małym ekranie tabele wychodzą poza viewport. Dla narzędzia monitoringowego, które może być sprawdzane na bieżąco, to krytyczne.

**3. Dostępność — formularze bez `<label for="">`**
Wszystkie `<label>` w `reading_form.html`, `investments.html`, `ev.html` nie mają atrybutu `for` powiązanego z `id` inputa. Screen readery nie powiążą labela z polem.

```html
<!-- Aktualnie w reading_form.html -->
<label style="...">☀️ Produkcja</label>
<input type="number" name="production_kwh" ...>

<!-- Powinno być -->
<label for="production_kwh" style="...">☀️ Produkcja</label>
<input id="production_kwh" type="number" name="production_kwh" ...>
```

### 🟡 Istotne

**4. Redundancja pól Rok/Miesiąc w `reading_form.html`**
Użytkownik wpisuje `period` (np. `2025.04`), a następnie osobno `year` i `month`. To trzy pola zamiast jednego — rok i miesiąc powinny być autouzupełniane z `period` przez prosty JS.

```html
<!-- Wystarczy jedno pole + auto-split po stronie JS/backend -->
<input type="month" name="period"> <!-- natywny picker YYYY-MM -->
```

**5. Komunikat po imporcie kasowany przez refresh**
W `import.html` sukces wyświetlany jest z query param:
```python
# main.py: RedirectResponse(url=f"/import?imported={n}&skipped={skipped}")
```
Odświeżenie strony ponownie pokazuje komunikat (lub go kasuje — zależy od zachowania przeglądarki). Wzorzec Post-Redirect-Get jest poprawny, ale komunikat powinien żyć w session flash, nie w URL.

**6. Brak wskaźnika ładowania przy imporcie pliku**
`import.html` — formularz z `<input type="file">` i `<button>`. Przetwarzanie Excela przez `openpyxl` może trwać chwilę. Brak żadnego feedbacku — użytkownik może kliknąć wielokrotnie.

```html
<!-- Prosta poprawa -->
<button type="submit" onclick="this.disabled=true; this.textContent='⏳ Importuję...'">
  📥 Importuj
</button>
```

**7. Nawigacja aktywna — false positives**
```html
<!-- base.html -->
{% if '/odczyty' in request.url.path %}active{% endif %}
```
URL `/odczyty/nowy` spełnia warunek dla "Odczyty" — poprawne. Ale `/odczyty/123/edytuj` też — to OK. Jednak ta sama logika dla `/roi` mogłaby kolidować z przyszłymi URL-ami zawierającymi `roi` w ścieżce. Lepiej stosować dokładne porównanie lub prefix check.

**8. Odkrywanie funkcji EV — słabe**
Pole `ev_kwh` w formularzu odczytu jest wizualnie ukryte po sekcji "Dodatkowe" i nie ma żadnego linku zwrotnego do `/ev`. Nowy użytkownik nie wie, że aplikacja śledzi samochody elektryczne dopóki nie wejdzie ręcznie w nav.

### 🟢 Drobne

**9. Hardkodowany `small` z oznaczeniami liczników**
```html
<th>Oddane <small style="color:#475569;">(2.8.0)</small></th>
```
Pojawia się w `readings.html`, `dashboard.html` i `reading_form.html` — każdorazowo osobno. Brak single source of truth dla tłumaczenia technicznego skrótu na label dla użytkownika.

**10. `chart.js` ładowany globalnie**
`base.html` ładuje Chart.js dla wszystkich stron, mimo że wykresy są tylko w `roi.html` i `dashboard.html`. Zbędny transfer dla stron `/import`, `/odczyty/nowy`.

## Rekomendacje

**1. [KRYTYCZNY] Dodaj obsługę błędów walidacji formularzy z powrotem do formularza**
Zwróć formularz z wypełnionymi danymi i komunikatem błędu zamiast cicho przekierowywać. W FastAPI/Jinja2: przekaż `errors: dict` do kontekstu szablonu.
*Biznesowo: błąd przy wprowadzaniu danych = utrata zaufania do narzędzia finansowego.*

**2. [WYSOKI] Napraw dostępność formularzy — `<label for="">`**
Zmiana w `reading_form.html`, `investments.html`, `ev.html`. 30 minut pracy, zero ryzyka regresji.
*Biznesowo: WCAG 2.1 AA to standard — narzędzia domowe też powinny go respektować.*

**3. [WYSOKI] Responsive tabele — `overflow-x: auto` wrapper**
Owiń każdą tabelę w `<div style="overflow-x:auto;">`. Chwilowa naprawa, docelowo — uproszczenie kolumn na mobile (ukrycie mniej istotnych przez media query).
*Biznesowo: monitoring z telefonu to naturalny use case dla narzędzia energetycznego.*

**4. [ŚREDNI] Autouzupełnianie Rok/Miesiąc z pola Okres**
Dodaj 5 linii JS w `reading_form.html`, które parsują `period` i wpisują `year`/`month`. Eliminuje błędy i redukuje liczbę wymaganych pól.

```javascript
document.querySelector('[name=period]').addEventListener('input', function() {
  const [y, m] = this.value.split('.');
  if (y) document.querySelector('[name=year]').value = y;
  if (m) document.querySelector('[name=month]').value = parseInt(m);
});
```

**5. [ŚREDNI] Loader przy imporcie pliku**
Jedno zdarzenie `onsubmit` na formularzu w `import.html`. Eliminuje wielokrotne kliknięcia i daje feedback przy dużych plikach Excel.

**6. [ŚREDNI] Guided onboarding dla pustego stanu**
Dashboard pusty stan (`dashboard.html`) pokazuje dwa przyciski. Dodaj trzeci krok: "Dodaj inwestycję → żeby śledzić ROI". Aktualnie użytkownik może mieć odczyty ale zero ROI, bo nie wie o sekcji Inwestycje.

**7. [NISKI] Lazy load Chart.js**
Przenieś `<script src="chart.js">` do bloków `{% block scripts %}` w szablonach które faktycznie używają wykresów (`roi.html`, `dashboard.html`). Oszczędność ~200KB transferu na pozostałych stronach.

**8. [NISKI] Ujednolicenie inline styles → klasy CSS**
`base.html` ma dobrze zdefiniowany mini design system (`.card`, `.btn-*`, `.stat-value`). Szablony powinny z niego korzystać zamiast powielać `style="font-size:1.5rem; font-weight:700"` w każdym `<h1>`. Sugestia: dodaj klasę `.page-title` do `base.html`.

## Sprzeczności w zespole

- **Security Architect vs Business Analyst** — Bezpieczeństwo: "Wieloużytkownikowość i autoryzacja to Won't Have — narzędzie single-tenant, dodanie auth zwiększyłoby złożoność bez wartości biznesowej". Rekomendacja jest oparta na założeniu niepotwierdzonym w kodzie.

1. **Jaki jest docelowy deployment?** Localhost na hoście domowym, Raspberry Pi w sieci LAN, czy serwer z dostępem zewnętrznym? Bez odpowiedzi połowa rekomendacji bezpieczeństwa jest albo zbędna, albo niewystarczająca.

Słuszna sprzeczność. `return TemplateResponse(422)` vs `return JSONResponse(422)` — te dwie ścieżki są niespójne. Moja rekomendacja była niekompletna.

**Korekta ADR-003:**

**ADR-003 (v2): Walidacja z obsługą błędów w warstwie HTML**
- **Kontekst:** Formularze zwracają HTML, nie JSON. HTTP 422 z Pydantic jako JSON jest bezużyteczny dla użytkownika przeglądarki.
- **Decyzja:** Walidacja w handlerze z re-renderingiem formularza przy błędach:

```python

async def create_reading(request: Request, ...):
    errors = {}
    
    # walidacja spójności period vs year/month (nowy zarzut krytyka!)
    if period != f"{year}.{month:02d}":
        errors["period"] = "Pole period niezgodne z rokiem/miesiącem"

        errors["sent_to_grid_kwh"] = "Oddane > Produkcja — fizycznie niemożliwe"
    
    if errors:
        return templates.TemplateResponse(
            "reading_form.html",
            {"request": request, "errors": errors, "data": form_data},
            status_code=422
        )
    # zapisz do DB
```

- **Konsekwencje:**
  - ✅ Błędy widoczne w UI bez JS
  - ✅ Walidacja spójności `period`/`year`/`month` — naprawia pominięty bug
  - ✅ Walidacja fizyczna `sent > production` — naprawia drugą lukę
  - ❌ Pydantic nie jest używany bezpośrednio (ale logika walidacji pozostaje w serwisie, testowalnie)

### ✅ Przyjmuję i rozszerzam: Spójność `period` vs `year`/`month`

Krytyk zidentyfikował realny bug, który **ja powinienem był wykryć** analizując schemat DB i formularz. Naprawia go ADR-003 v2 powyżej.

**Dodatkowa rekomendacja:** Usunąć redundancję na poziomie schematu:

```sql
-- Opcja A: wyliczaj year/month z period (usuwa redundancję)
CREATE VIEW readings_v AS
    SELECT *, 
        CAST(substr(period, 1, 4) AS INTEGER) as year,
        CAST(substr(period, 6, 2) AS INTEGER) as month
    FROM readings;

-- Opcja B: DB constraint (zachowuje kolumny dla wydajności indeksów)
-- CHECK (year = CAST(substr(period, 1, 4) AS INTEGER) 
--   AND month = CAST(substr(period, 6, 2) AS INTEGER))
```

Rekomenduję **Opcję B** — constraint w SQLite jest tani, a kolumny `year`/`month` są używane w `ORDER BY year, month`.

Token HA w plaintext w SQLite = token HA w plaintext w chmurowym backupie. Cena mitygacji (jeden import `cryptography`, jeden klucz w `.env`) jest znikoma względem ryzyka przejęcia smart home.

**2. Sprzeczność z Business Analysteml — wymagam rozstrzygnięcia**
